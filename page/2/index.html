发表于 2017-04-17   |     |   阅读次数

JVM方法调用的内部机制

栈帧结构

为了调用方法和返回方法,JVM维护了一个栈结构,里面存放的实体称为栈帧,每个栈帧包括 局部变量表 操作数栈 动态链接 方法返回地址 和额外的信息.在编译期,栈帧需要多大的局部变量表已经确定了

局部变量表

存放方法参数和方法内创建的局部变量,局部变量表的容量以 Slot 为最小单位,一个 Slot 可以存放一个 32位以内的数据类型.long double 需要两个

如果方法时非 static 的,那么局部变量表中第0位索引 Slot 默认用于传递方法所属对象实例的引用 (this)

为了节省栈帧空间,Slot是可以重用的,一个局部变量定义了没有赋初始值是不能用的

操作数栈

JVM解析执行字节码是基于栈结构,调用其他方法是通过操作数栈来进行参数传递的

方法大致调用过程
  • 每一次调用指令前,JVM把方法被调用的对象引用引入操作数栈,JVM还会把方法的参数依次放入操作数栈
  • 调用方法的时候,JVM将参数和对象引用依次从操作数栈弹出,并建立一个新的栈帧存放刚才弹出的变量放入 Slot 1,2,3..
  • JVM把新栈帧push入虚拟机方法栈,并把PC指向函数第一条待执行的指令

JVM通过编译器的静态绑定或者运行时期的动态绑定获取被调用方法的地址

方法调用字节码指令

  • invokestatic 调用静态方法
  • invokespecial 调用实例构造器,私有方法和父类方法
  • invokeVirtual 调用所有的虚方法(静态方法,私有方法,实例构造器,父类方法,final 方法 都不是 虚方法)
  • invokeinterface 调用接口方法,在运行时期确定一个实现此接口的对象

invokestatic 和 invokespecial 在类加载阶段就把符号解析为该方法的引用,静态

invokevirtual和invokeinterface 是在运行时确定具体的方法地址

动态分配

当JVM 遇到 invokevirtual或者invokeinterface时,需要运行时根据方法符号找到方法地址,具体过程如下

  • 方法调用前,将对象的引用压入操作数栈
  • 方法调用时,找到操作数栈顶的第一个元素所指向的对象实际类型,记为C
  • 在 C类中找到常量池中描述符和方法名都相符的方法,并校验访问权限,找到方法并通过校验,就返回这个方法的引用
  • 没找到继续按照继承关系向上找并且校验权限
  • 还是没找到 抛出AbstractMethodError异常

为了提高动态分配时候方法查找的效率,JVM为每个类维护一个虚函数表

虚函数表

目的:为了快速找到方法索引地址

JVM在链接类的过程中,给每个类分配相应方法表内存空间,每个类对应一个方法表,都存在于 方法区 中,Java 每个对象的对象头有一个类型指针,可以找到对应的类,类数据中对应一个方法表,类级别的方法表

方法表包含了类所有方法的入口地址,父类继承的方法放在前面,然后是**接口

invokevirtual 与 invokeInterface 区别

invokevirtual虚方法是按照父类到子类的顺序排序的,因此使用 invokevirtual 调用虚函数,JVM可以再编译器就确定虚函数在方法表上的 offset ,从而快速定位方法地址

invokeInterface 接口可以由不同的类实现,所以接口方法在不同类的中 方法表 offset 都不一样,因此每次调用接口方法,都要搜寻一遍虚函数表,效率低于 invokevirtual

JVM学习笔记(三):Java内存模型

发表于 2017-04-10   |     |   阅读次数

java 内存模型

目标:定义程序各个变量的访问规则

  • 主内存(共享->java堆): 所有变量都存储在这
  • 工作内存(私有-虚拟机栈): 变量在工作内存中只是拷贝的副本

内存间交互操作

java内存模型定义了以下八种操作,都是原子性的

  • lock(锁定)
  • unlock(解锁)
  • read(读取)
  • load(载入)
  • use(使用)
  • assign(赋值)
  • store(存储)
  • write(写入)

使用八种操作要符合以下规则

  • 一个新变量只能在主内存中诞生,对一个变量 use,store前,必须 assign和load
  • 对一个变量 lock 了,那么会清空 工作内存 中变量的值,然后需要重新load或assign
  • 对一个变量 unlock 之前,必须把变量同步回主内存中,store,write

volatile 操作

  • 保证了可见性原理
    • 产生一个内存屏障:他使用一个lock操作,这样会清空工作内存的值,然后从新执行load assign,获取主内存的最新数据
    • lock后会unlock,会执行 store,write,将数据写回主内存,通过这两点保证了可见性
  • 保证了有序性:
    • 因为内存屏障,lock指令会把修改同步到主内存中,这样形成了指令重排序无法越过内存屏障也就是volatile

原子性

  • 八个基本操作是原子性的,基本数据类型的访问读写是具有原子性的
  • 更大范围需要保证原子性,可以用synchronized

可见性

一个线程修改了共享变量的值,其他线程能立即获取到这个修改

  • synchronized 通过 unlock 之前,必须将变量同步回主内存
  • final 通过 不传递 this

有序性

  • volatile 通过禁止指令重排序保证有序性
  • synchronized 通过 一个变量同一时刻只能有一个线程 lock

先行先发生 hapends-before

如果两个操作间的关系不在下面规则内,那么就没有顺序性保障

  • 程序次序规则:单个线程内程序流顺序
  • 官程锁定规则:对于同一个锁,unlock发生在lock之前
  • valatile 变量规则
  • 线程启动规则:start()先行于线程内任何操作
  • 线程终止规则:所有操作先行于 线程终止
  • 线程中断规则:interrupt()先行发生于被中断的代码
  • 对象终结规则:对象的初始化先行于 finalize()
  • 传递性

JAVA 与 线程

线程的实现

  • 使用内核线程实现
  • 使用用户线程实现
  • 使用用户线程+轻量级进程(线程)实现

    线程调度

  • 协同式调度:执行时间由线程本身控制
  • 抢占式调度,由系统分配执行时间

JVM学习笔记(二):垃圾回收

发表于 2017-04-07   |     |   阅读次数

对象已死吗?

  • 引用计数法:有引用就+1,但是两个对象互相持有引用的时候就没办法回收
  • 可达性分析:通过一系列 GC Root作为起始点,向下搜索,走过的路称为引用链,当一个对象到GC Root没有任何引用链时,就成为不可用的,什么可以当做 GC Root?
    • 虚拟机栈中的引用对象
    • 方法区中静态属性引用的对象
    • 常量引用的对象
    • JNI引用的对象

对象生存还是死亡?

  • 当对象没有 GC Root 引用时,会被第一次标记,然后进行筛选,筛选的条件是对象有没有必要执行 finalize() 方法,对象没有覆盖finalize()方法或者已经调用过都算没有必要执行
  • 如果有必要执行,对象会被放置在 F-Queue队列中,然后被一个低优先级线程执行回收.

垃圾回收算法

标记-清除算法

过程:标记处需要回收的对象,标记完成后统一回收.

缺点:效率低,空间碎片

复制算法

复制算法是先将内存分为连个部分，一部分用来放入对象，而另一部分暂时不用，当使用的一部分内存要进行垃圾回收的时候会将不需要回收的对象复制保存在另一个空间中，然后再对使用过的那部分区域进行垃圾回收，这样虽然效率很高，但是很浪费空间，

所以一般将新生代分为Eden空间和两个Survivor空间，其大小在HotSpot虚拟机中默认比例为8：1：1，这样在新生代中采用复制算法回收垃圾效率就很高了.

具体回收过程是将Eden区域和From Survivor区域作为对象的存储空间，当要进行垃圾回收的时候先将这两个区域中不需要回收的对象复制保存在To Survivor区域中，然后再进行垃圾回收,两块 Survivo 交换作用。另外有一点是当一个对象在Eden区域和From Survivor区域中复制到To Survivo的时候发现内存不足，这时会进行内存 分配担保

标记 - 整理算法

与 标记-清除 一样,只是后续步骤让所有存活对象都向一端移动,然后清除边界外的内存

分代算法

java区分为新生代和老年代,

  • 新生代用 复制算法

  • 老年代用 标记-清除, 标记-整理

内存分配与回收策略

  • 对象优先在 Eden 区域分配, Eden 区域不够时,进行一次 Minor GC:时间短，频率高
  • 大对象直接进老年代,老年代内存不够进行 Full GC时间长，频率低

  • 长期存活的对象进入老年代(解释如下)

    • 关于对象的分配：对象优先在Eden区域分配，大对象会直接进入老年代，长期存活的对象会进入老年代，这里的长期存活是根据新生代中的对象年龄阈值来定义的，对象刚分配到新生代的时候年龄为1，每进行一次GC对象的年龄会加1，HotSpot中默认的阈值是15，也就是说对象年龄达到15岁的时候会被分配到老年区，这个值是可以通过参数配置的。
    • 如果相同年龄中所有对象大小总和大于 Survivo的一半,那么大于等于这个相同年龄的对象都进入老年代

空间分配担保

  • Minor GC 前,虚拟机会检查老年代最大可连续用空间是否大于新生代所有对象空间,大于则 Minor GC是安全的
  • 如果不大于,那么看是否允许设置担保失败,允许话:检查老年代最大可以的连续空间是否大于历次晋级到老年代的对象的平均大小,大于则尝试 Minor GC,小于的话 Full GC

垃圾收集器

  • Serial 收集器:(新生代)单线程收集器,垃圾回收时暂停其他所有工作线程Stop the world
  • ParNew 收集器:(新生代)多线程版本的 Serial,其他都一样
  • Parallel Scavenge 收集器:(新生代)为了达到可控制的吞吐量,而不是缩短停顿时间
  • Serial Old 收集器:(老年代) 单线程 Serial 收集器的老年代版本
  • Parallel Old 收集器:(老年代) 多线程 注重 吞吐量
  • CMS 收集器:以获取最短回收停顿时间为目标,采用 标记-清除算法
    • 初始标记 -> 并发表及 -> 重新标记- > 并发清楚
    • 缺点:对CPU资源敏感 -> 无法清除浮动垃圾- > 标记-清除,产生空间碎片

JVM学习笔记(一):内存分配

发表于 2017-04-03   |     |   阅读次数

java堆

存放对象实例,几乎所有对象实例都在这里创建,垃圾收集器的主要区域(线程共享)

方法区

  • 存储 类信息,常量,静态变量,编译后的代码等 (线程共享)
  • 运行时常量池:存储编译期产生的字面量和符号引用, 类加载后放入运行时常量池

JVM虚拟机栈

java 方法的内存模型,用于存储局部变量表、操作数栈、动态链接、方法出口等信息。每个方法从调用直至执行完成的过程，就对应着一个栈帧在虚拟机栈中入栈到出栈的过程。(线程私有)

栈可能会出现栈溢出

本地方法栈

本地方法栈和JVM栈非常相似，它们之间的区别不过是jvm栈是为执行java方法服务，而本地方法栈是为jvm使用到对的本地方法服务。HotSpot虚拟机中直接把本地方法栈和JVM栈合二为一了。(线程私有)

程序计数器

看做当前程序执行字节码的行号指示器(如果本地方法就是空的) 通过改变这个计数器选择下一条需要执行的字节码指令,分支\循环\跳转\异常处理\线程回复 ,唯一没有 OutOfMemory的区域. (线程私有)

对象的内存布局

  • 对象头:存储对象自身的运行时数据
    • 哈希码,GC分代年龄,锁状态标志,
    • 类型指针,指向它的类元数据的指针,听过这个指针确定是哪个类的实例
    • 如果是 java 数组,还有一块记录数据长度的数据
  • 实例数据
  • 对齐填充

对象的创建过程

  1. 先检查常量池中能否定位到一个类的符号引用,并且检查类是否已经加载,解析和初始化过了.
  2. 为对象分配内存,在 java 堆中,如果 java 堆 规整,那么通过 指针碰撞 分配出内存空间.否则利用 空闲列表找到足够大的空间划分给对象。需要的内存大小在类加载的时候已经确定了。

指针碰撞：java 内存规整，所有使用过得内存放一边，没有使用过的放在另一边，中间有一个指针作为分界点。分配内存就是把指针指向空闲空间移动一段与内存大小相等的距离。

空闲列表：Java 内存不规整，虚拟机维护一张表，记录那块内存可用。再分配时在列表中找到一块足够大的空间划分给对象实例。并更新列表记录。

  1. 内存分配完后,JVM将内存空间的初始化零值,然后对于 对象头,进行设置(所属类实例,对象哈希码,对象GC分代年龄)
  2. 执行,将对象按照代码意愿进行初始化,对象创建.

对象的内存布局

  1. 对象头：

    1.1 ：第一部分用于存储对象自身的运行时数据，如哈希码（HashCode）、GC分代年龄、锁状态标志、线程持有的锁、偏向线程ID、偏向时间戳等。

    1.2：类型指针，即对象指向它的类元数据的指针，虚拟机通过这个指针来确定这个对象是哪个类的实例。

    1.3：如果对象是一个数组，那么还得记录数组的长度。

  2. 实例数据：对象真正存储的有效信息，程序中定义的各中字段

  3. 对齐填充：将对象大小填充为8的整数倍

对象的访问定位

  • 使用句柄: java划分一块内存作为句柄池,引用存的是对象的句柄地址,然后句柄再指向对象地址
  • 直接指针: 引用直接指向对象地址

Android Bitmap 占多大内存

发表于 2017-03-05   |     |   阅读次数

在 Android 中，通过 BimapFctory.Options 来加载所需尺寸的图片，它包含四种色彩存储方法

  • ALPHA_8 ： alpha 由8位组成 ，那么一个像素占 8位
  • ARGB_4444 ：alpha 4位， red 4位， green 4位， blue 4位， 那么一个像素占 4*4 = 16位
  • ARGB_8888 ： alpha 8位， red 8位， green 8位， blue 8位， 那么一个像素点占 32位
  • RGB_565 ： red 5位， green 6位， blue 5位，那么一个像素点占 16位

图的位数越高，占用的内存越大

一张图片占用的内存取决于

  • 图片色彩模式
  • 原始资源的density（资源目录）
  • 手机的dp

科普:资源文件对应的dpi

  • ldpi 120dpi
  • mdpi 160dpi
  • hdpi 240dpi
  • xhdpi 320dpi
  • xxhdpi 480dpi

那么,给定一张图片,他在手机内存中所占大小为:

= (imageWidth/density*手机dp+0.5) * (imageHeigth/density*手机dp+0.5) * 一个像素所占字节

比如,一张 522 * 686 的 PNG (一个像素4字节 )图片,把它放到 drawable-xxhdpi 下,在三星S6(640dp)上加载,所占内存为:

(522/480 640 +0.5) (686/480 640+0.5) 4 = 2547360

dp = px*160/dpi

所以对于 160dpi的屏幕 1dp = 1px

可以看到，SerialExecutor是使用ArrayDeque这个队列来管理Runnable对象的，如果我们一次性启动了很多个任务，首先在第一次运行execute()方法的时候，会调用ArrayDeque的offer()方法将传入的Runnable对象添加到队列的尾部，然后判断mActive对象是不是等于null，第一次运行当然是等于null了，于是会调用scheduleNext()方法。在这个方法中会从队列的头部取值，并赋值给mActive对象，然后调用THREAD_POOL_EXECUTOR去执行取出的取出的Runnable对象。之后如何又有新的任务被执行，同样还会调用offer()方法将传入的Runnable添加到队列的尾部，但是再去给mActive对象做非空检查的时候就会发现mActive对象已经不再是null了，于是就不会再调用scheduleNext()方法。

那么后面添加的任务岂不是永远得不到处理了？当然不是，看一看offer()方法里传入的Runnable匿名类，这里使用了一个try finally代码块，并在finally中调用了scheduleNext()方法，保证无论发生什么情况，这个方法都会被调用。也就是说，每次当一个任务执行完毕后，下一个任务才会得到执行，SerialExecutor模仿的是单一线程池的效果

view有三个重要的方法：

  • dispatchTouchEvent 是否消耗当前事件，消耗返回true
  • onIterceptTouchEvent 是否拦截，拦截了的话就不下发事件，拦截返回true
  • onTouchEvent 返回结果表示是否消耗了当前事件
    • onTouchListener 的 onTouch
      • OnClickListener 的 onClick

第一段 if 走完后，如果 onTouch 返回 false，我们将进入 onTouchEvent 方法，由于 result 默认为 false ，那么 第二条if判断中， dispatchTouchEvent 的值取决于 onTouchEvent 的返回值，那我们看看 onTouchEvent 的大致源码.

  • view 中事件的分发的顺序是 dispatchOnTouchEvent -> OnTouchListener 的 onTouch 方法，如果返回 false -> onTouchEvent -> onClickListener 的 onClick

经过测试后可以小结:

  • onTouchEvent 返回结果影响 dispathTouchEvent 事件，onTouchEvent 返回 true 则 dispatchTouchEvent 返回 true，反之。

ViewGroup 的分发机制

  • onInterceptTouchEvent ( if (!canceled && !intercepted) 返回false，这样就无法分发给子 view )表示是否对事件进行拦截，返回true 则拦截，反之。默认返回 false.

  • 子View中如果将传递的事件消费掉(dispatchTouchEvent 返回 false，从而导致 dispatchTransformedTouchEvent 返回 false )，导致 没执行 addTouchTarget ，这样 mFirstTouchTarget 依旧为 null，那么 ACTION-MOVE , ACTION-DOWN 就无法执行了。

总结

在view 中

  • view点击事件的顺序是
    • OnTouchListener 的 onTouch
    • 如果 onTouch 返回 false，则执行 onTouchEvent
    • 在 onTouchEvent 中，如果前一个action返回false，那么后面的action都不会执行
  • view 的enable 属性不影响 onTouchEvent 的默认返回 true，只要他的 clickable ，longClickable 有一个为 true，那么他的 onTouchEvent 就返回 true。

在 ViewGroup 中

  • 点击事件先传到 viewGroup ，然后再传到 子view
  • viewGroup 中有一个 onInterceptTouchEvent（）方法，如果该方法返回true则将事件拦截，子view无法处理事件。返回false则不拦截，默认返回 false
  • viewGroup 一旦决定拦截该事件，那么事件都交给它处理，而且不再调用 onInterceptTouchEvent() 方法（mFirstTouchTarget 为 null，不再进入 true 代码块）.
  • 如果子view消耗了事件，那么viewGroup将无法接收到事件。如果所有子view都无法处理事件，那么将回交给viewGroup处理。
