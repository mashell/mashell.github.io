JVM学习笔记(一):内存分配

发表于 2017-04-03   |     |   阅读次数

java堆

存放对象实例,几乎所有对象实例都在这里创建,垃圾收集器的主要区域(线程共享)

方法区

  • 存储 类信息,常量,静态变量,编译后的代码等 (线程共享)
  • 运行时常量池:存储编译期产生的字面量和符号引用, 类加载后放入运行时常量池

JVM虚拟机栈

java 方法的内存模型,用于存储局部变量表、操作数栈、动态链接、方法出口等信息。每个方法从调用直至执行完成的过程，就对应着一个栈帧在虚拟机栈中入栈到出栈的过程。(线程私有)

栈可能会出现栈溢出

本地方法栈

本地方法栈和JVM栈非常相似，它们之间的区别不过是jvm栈是为执行java方法服务，而本地方法栈是为jvm使用到对的本地方法服务。HotSpot虚拟机中直接把本地方法栈和JVM栈合二为一了。(线程私有)

程序计数器

看做当前程序执行字节码的行号指示器(如果本地方法就是空的) 通过改变这个计数器选择下一条需要执行的字节码指令,分支\循环\跳转\异常处理\线程回复 ,唯一没有 OutOfMemory的区域. (线程私有)

对象的内存布局

  • 对象头:存储对象自身的运行时数据
    • 哈希码,GC分代年龄,锁状态标志,
    • 类型指针,指向它的类元数据的指针,听过这个指针确定是哪个类的实例
    • 如果是 java 数组,还有一块记录数据长度的数据
  • 实例数据
  • 对齐填充

对象的创建过程

  1. 先检查常量池中能否定位到一个类的符号引用,并且检查类是否已经加载,解析和初始化过了.
  2. 为对象分配内存,在 java 堆中,如果 java 堆 规整,那么通过 指针碰撞 分配出内存空间.否则利用 空闲列表找到足够大的空间划分给对象。需要的内存大小在类加载的时候已经确定了。

指针碰撞：java 内存规整，所有使用过得内存放一边，没有使用过的放在另一边，中间有一个指针作为分界点。分配内存就是把指针指向空闲空间移动一段与内存大小相等的距离。

空闲列表：Java 内存不规整，虚拟机维护一张表，记录那块内存可用。再分配时在列表中找到一块足够大的空间划分给对象实例。并更新列表记录。

  1. 内存分配完后,JVM将内存空间的初始化零值,然后对于 对象头,进行设置(所属类实例,对象哈希码,对象GC分代年龄)
  2. 执行,将对象按照代码意愿进行初始化,对象创建.

对象的内存布局

  1. 对象头：

    1.1 ：第一部分用于存储对象自身的运行时数据，如哈希码（HashCode）、GC分代年龄、锁状态标志、线程持有的锁、偏向线程ID、偏向时间戳等。

    1.2：类型指针，即对象指向它的类元数据的指针，虚拟机通过这个指针来确定这个对象是哪个类的实例。

    1.3：如果对象是一个数组，那么还得记录数组的长度。

  2. 实例数据：对象真正存储的有效信息，程序中定义的各中字段

  3. 对齐填充：将对象大小填充为8的整数倍

对象的访问定位

  • 使用句柄: java划分一块内存作为句柄池,引用存的是对象的句柄地址,然后句柄再指向对象地址
  • 直接指针: 引用直接指向对象地址

Android Bitmap 占多大内存

发表于 2017-03-05   |     |   阅读次数

在 Android 中，通过 BimapFctory.Options 来加载所需尺寸的图片，它包含四种色彩存储方法

  • ALPHA_8 ： alpha 由8位组成 ，那么一个像素占 8位
  • ARGB_4444 ：alpha 4位， red 4位， green 4位， blue 4位， 那么一个像素占 4*4 = 16位
  • ARGB_8888 ： alpha 8位， red 8位， green 8位， blue 8位， 那么一个像素点占 32位
  • RGB_565 ： red 5位， green 6位， blue 5位，那么一个像素点占 16位

图的位数越高，占用的内存越大

一张图片占用的内存取决于

  • 图片色彩模式
  • 原始资源的density（资源目录）
  • 手机的dp

科普:资源文件对应的dpi

  • ldpi 120dpi
  • mdpi 160dpi
  • hdpi 240dpi
  • xhdpi 320dpi
  • xxhdpi 480dpi

那么,给定一张图片,他在手机内存中所占大小为:

= (imageWidth/density*手机dp+0.5) * (imageHeigth/density*手机dp+0.5) * 一个像素所占字节

比如,一张 522 * 686 的 PNG (一个像素4字节 )图片,把它放到 drawable-xxhdpi 下,在三星S6(640dp)上加载,所占内存为:

(522/480 640 +0.5) (686/480 640+0.5) 4 = 2547360

补充

dp dip sp px dpi 区别与转换

px: 像素

dpi:屏幕每英寸所包含像素点数目

dpi = 屏幕对角线像素/ 屏幕英寸

dp == dip :密度无关单位

px = dp*dpi/160

dp = px*160/dpi

所以对于 160dpi的屏幕 1dp = 1px

但是实际的dpi是固定的,120,160,240,320,480

不同屏幕分辨率下dp差异,我们可以通过单独适配的方法:比如对 360dp布局做适配,将布局文件放在 layout-w320dp 中,相对单独分辨率做适配,就放在标有分辨率的文件夹中=> layout-854x480

  • onInterceptTouchEvent ( if (!canceled && !intercepted) 返回false，这样就无法分发给子 view )表示是否对事件进行拦截，返回true 则拦截，反之。默认返回 false.

总结

在view 中

  • view点击事件的顺序是
    • OnTouchListener 的 onTouch
    • 如果 onTouch 返回 false，则执行 onTouchEvent
    • 在 onTouchEvent 中，如果前一个action返回false，那么后面的action都不会执行
  • view 的enable 属性不影响 onTouchEvent 的默认返回 true，只要他的 clickable ，longClickable 有一个为 true，那么他的 onTouchEvent 就返回 true。

在 ViewGroup 中

  • 点击事件先传到 viewGroup ，然后再传到 子view
  • viewGroup 中有一个 onInterceptTouchEvent（）方法，如果该方法返回true则将事件拦截，子view无法处理事件。返回false则不拦截，默认返回 false
  • viewGroup 一旦决定拦截该事件，那么事件都交给它处理，而且不再调用 onInterceptTouchEvent() 方法（mFirstTouchTarget 为 null，不再进入 true 代码块）.
  • 如果子view消耗了事件，那么viewGroup将无法接收到事件。如果所有子view都无法处理事件，那么将回交给viewGroup处理。

java 九种单例模式

发表于 2017-01-10   |     |   阅读次数

单例模式大家都很常见了，我这篇文章是看了别人的文章后自己学习后写出来的，相当于一个笔记吧。

饿汉式

  • 优点：线程安全

  • 缺点：由于类加载机制，无论类是否被使用，类第一次加载进入内存就回实例化

饿汉式 – 静态代码块

  • 优点:线程安全

  • 缺点：类装载时创建实例，若实例一直未被使用时，会浪费资源

懒汉式

  • 优点：避免了第一次加载这个类进入内存未使用占用内存

  • 缺点：线程不安全,当多线程情况下 getInstance时可能会生成多个实例

懒汉式 – 方法同步

  • 优点：线程安全

  • 缺点：同步方法，实际上只需要处理 new Singleton 的同步问题即可，效率太低了

懒汉式 – 代码块同步

  • 优点：效率高于 方法同步 的懒汉式

  • 缺点：判断 singleton 是否为空的时候，如果此时多线程访问了，那么也存在异步问题，实例化了多个singleton

懒汉式 – 错误的 double check

  • 优点：效率高

  • 缺点：线程不安全，依然存在创建多个实例的可能。因为每个线程都有自己的一份拷贝，并不能保证实例化后将INSTANCE的引用拷回主内存，不能保证对其它线程立即可见，所以仍然有可能造成多个实例被创建，具体原因搜索 volatile 关键字

懒汉式 – 正确的 doucle check

  • 优点：效率高，解决了线程不安全的问题

枚举

  • 优点: 无线程同步问题，且能防止通过反射创建新的对象

  • 缺点: 使用的是枚举，而非类。同时单一实例的访问点也不是一般单例模式的静态方法

静态内部类

  • 优点： 静态内部类不存在类加载的时候就初始化，当加载内部类的时候，java 虚拟机初始化其成员变量并且保证线程安全，而且没有锁不会造成性能影响

总结

以上就是我的笔记，总的来说一般我常用静态内部类的模式，最终的模式选择取决于大家。

Android 内存泄露解析

发表于 2016-12-15   |     |   阅读次数

内存泄露也是一个老生常谈的问题了，我们今天来学习怎样会造成内存泄露以及如何避免内存泄露，内存泄露该如何解决。

怎样造成内存泄露的？如何避免它？

Android虚拟机的垃圾回收采用的是根搜索算法。GC会从根节点（GC Roots）开始对heap进行遍历。到最后，部分没有直接或者间接引用到GC Roots的就是需要回收的垃圾，会被GC回收掉。而内存泄漏出现的原因就是GC对象的根存在了无效的引用，导致本来需要被GC的对象没有被回收掉。

简单来说就是当一个对象需要释放内存时，他被其他生命周期长的对象所引用，导致JVM无法回收这个对象的内存，从而导致了内存泄露。

主要有三种情况

  • 静态变量
  • 非静态内存类
  • RxJava 忘记取消注册引起的内存泄露（实际上也是第二点的原因）
  • 无限进行的动画

其他情况:

  • 资源对象没关闭。如Cursor、File等资源。他们会在finalize中关闭，但这样效率太低。容易造成内存泄露。SQLiteCursor，当数据量大的时候容易泄露
  • 广播注册了没有取消造成内存泄漏

1,静态变量造成的内存泄露

原因：

在这一个例子中，我们的 LeakActivity 被静态变量 Context 所持有，我们知道静态变量的生命周期是与整个应用的生命周期相同的(如果这方面知识点不太清楚可以看看这里 Android 静态生命周期，那么当我们 LeakActivity 退出需要被回收时，垃圾回收机制发现 LeakActivity 被一个静态变量持有引用，那么导致无法回收 LeakActivity 的内存导致内存泄露。

解决办法：

在 onDestory 方法中将 静态变量 sContext 的值 置为 null，解除对 Activity 的引用。

使用 Application 的context 传入 Singleton ，Application的生命周期是最长的，它的生命周期就等于这个程序的生命周期，这样就不会导致内存泄露了。

2，非静态内存类持有外部类应用导致的内存泄露

原因：

内存类会持有外部类的一个引用，在这个例子中，Handler 创建了一个匿名内部类，那么这个内部类就会持有外部类的引用，而静态内部类不会。而我们知道 Handler 是一直有 Looper 在 MessageQueue 里面取消息的，如果有一个消息一直没取出来，那么 Mesasge 有 Handler 的引用， Handler 有 Activity 的引用，这样最终导致内存泄露。

或者其他情况，在内部类中，如果开启一个线程执行一项耗时任务，在执行任务时打算销毁外部类时，垃圾回收机制检测到Activity被内部类持有引用，同样导致无法回收 LeakActivity 。

解决办法：

使用弱引用 WeakReference + 静态内部类

或者内部类是静态的，这样就不会持有外部类引用了。当然，用了静态内部类后就注意 知识点1 的内容，别自己手动导致静态内部类持有activity的引用了。

总结一下

  • Message：消息，其中包含了消息ID，消息处理对象以及处理的数据等，由MessageQueue统一列队，终由Handler处理。
  • Handler：处理者，负责Message的发送及处理。使用Handler时，需要实现handleMessage(Messagemsg)方法来对特定的Message进行处理，例如更新UI等。
  • MessageQueue：消息队列，用来存放Handler发送过来的消息，并按照FIFO规则执行。当然，存放Message并非实际意义的保存，而是将Message以链表的方式串联起来的，等待Looper的抽取。
  • Looper：消息泵，不断地从MessageQueue中抽取Message执行。因此，一个MessageQueue需要一个Looper。
  • Thread：线程，负责调度整个消息循环，即消息循环的执行场所。

至此，我们看到，一个Message经由Handler（在其他线程）的发送，MessageQueue的入队，Looper的抽取，又再一次地回到Handler的怀抱。而绕的这一圈，也正好帮助我们将异步操作变成了同步操作。

Q3 ThreadLocal 是什么

ThreadLocal是一个关于创建线程局部变量的类。
通常情况下，我们创建的变量是可以被任何一个线程访问并修改的。而使用ThreadLocal创建的变量只能被当前线程访问，其他线程则无法访问和修改。

所以在Looper对象中通过sThreadLocal就可以找到其绑定的线程。ThreadLocal中有个set方法和get方法，可以通过set方法向ThreadLocal中存入一个对象，然后可以通过get方法取出存入的对象。ThreadLocal在new的时候使用了泛型，从上面的代码中我们可以看到此处的泛型类型是Looper，也就是我们通过ThreadLocal的set和get方法只能写入和读取Looper对象类型，如果我们调用其ThreadLocal的set方法传入一个Looper，将该Looper绑定给了该线程，相应的get就能获得该线程所绑定的Looper对象。
